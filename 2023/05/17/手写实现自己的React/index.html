一旦我们开始渲染，我们将不会停止，直到我们渲染完完整的元素树。我们可以想象，如果这个 Element tree 足够大，它可能会阻塞主线程太久，如果浏览器需要做高优先级的事情，比如处理用户输入或保持动画流畅，它必须等到渲染完成。

此时，需要一个解决方案：

我们可以把工作分解成小的单元，当我们完成每个单元后，如果还有其他高优先级的事情，比如处理用户输入或保持动画流畅，我们会让浏览器中断渲染，优先去处理高优先级的工作。

window.requestIdleCallback()方法插入一个函数，这个函数将在浏览器空闲时期被调用。这使开发者能够在主事件循环上执行后台和低优先级工作，而不会影响延迟关键事件，如动画和输入响应。

这样即可实现我们的需求：暂停渲染去优先处理其他高优先级的事情。

但是，还会有一个问题，我们在暂停渲染并处理完其他高优先级的工作后，后续又是如何接着上次渲染的工作单元，从而继续渲染的呢，维持这样的工作需要什么样的数据结构呢？

3、Fibers

为了组织工作单元，我们需要一个数据结构：fiber tree。

我们将为每个 Element 提供一个 fiber，每个 fiber 将成为一个工作单元。

举个例子，假设我们要渲染这样一个元素树：

其 fiber tree 为下图所示：

在渲染中，我们将创建root fiber并将其设置为工作单元 nextUnitOfWork，剩下的工作将发生在 performUnitOfWork 函数上，我们将为每个 fiber 做三件事：

  • 将元素添加到 DOM
  • 为元素的子元素创建 fiber
  • 选择下一个工作单元

该数据结构的目标之一是便于查找下一个工作单元。这就是为什么每个 fiber 都有一个链接到它的第一个 child，它的下一个 sibling 和它的 parent。

那么我们如何查找下一个工作单元，即下一个要处理的 fiber 呢，流程如下：

  • 当我们完成对 fiber 的工作时，如果它有一个 child fiber，那么该 fiber 将成为下一个工作单元。在我们的示例中，当我们完成对 div fiber 的处理后，下一个工作单元将是 h1 fiber。
  • 如果 fiber 没有 child，我们使用其 sibling 作为下一个工作单元。例如，p fiber 没有 child，所以我们在完成后移动到 a fiber。
  • 如果 fiber 既没有 child 也没有 sibling，我们就去找“叔叔”：即 parent 的 sibling。就像示例中的 a 和 h2 fiber。
  • 如果 parent 没有 sibling，我们会继续向上遍历 parent，直到我们找到一个有 sibling 的 parent 或直到我们到达 root。如果我们已经到达 root，则意味着我们已经完成了此渲染的所有工作。

二、问题解决

1、排查问题

  1. 根据上述接口报错跨域截图，第一印象可知，其报错原因可能是跨域，观察可知其接口域名https://cheyipai.com和接口请求所在的页面 url 地址https://souche.com，但和后端同学沟通得到，服务端是对该页面的 url 域名做过 CORS 配置处理的，所以理论上不会被服务器因为跨域而拦截的。

  2. 但是浏览器控制台是明确报错跨域的，再查看上述接口请求信息可知，其请求头是缺少Origin字段的，所以是不是可能因为缺少Origin字段，导致后端 CORS 规则识别匹配失败，进而被跨域拦截。

  3. 查看资料HTTP 标头 - Origin，了解如下：

    • 浏览器会在以下情况中添加 Origin 请求标头：

      • 跨源请求。
      • 除 GET 和 HEAD 以外的同源请求（即它会被添加到同源的 POST、OPTIONS、PUT、PATCH 和 DELETE 请求中）。
      • 除上述规则外，还有一些特殊情况。例如，在 no-cors 模式下的跨源 GET 或 HEAD 请求不会发送 Origin 标头。

    • Origin 标头在以下情况中（不完整）会被设置为 null：

      • 请求来源的协议不是 http、https、ftp、ws、wss 或 gopher 中的任意一个（如：blob、file 和 data）。
      • 跨源的图像或媒体，包括：img、video 和 audio 元素。
      • 属于以下几种文档类型的：使用 createDocument() 创建的、通过 data: URL 生成的或没有创建者的浏览上下文的。
      • 跨源重定向。
      • 没有为 sandbox 属性设置 allow-same-origin 值的 iframe。
      • 响应（response）是网络错误。

      由上可知，在浏览器中导致 HTTP 请求丢失Origin标头的原因可能为：响应（response）是网络错误。

      那么，什么情况下会导致网络错误呢？可以先回想下 OSI 模型，如下图所示：

      我们知道，HTTP 请求的过程：DNS 解析 -> TCP 三次握手 -> 发起 HTTP 请求 -> 服务器响应 HTTP 请求 -> 浏览器解析-> 浏览器进行页面渲染 -> 服务器关闭 TCP 连接。

      那么根据上述接口响应信息可知，该接口并无响应，可以判断该 HTTP 请求问题可能出在TCP 已经连接并 HTTP 请求发送后的某个阶段。

      那么如何知道该 HTTP 请求是否 TCP 连接成功呢？

  4. 再查看资料调试网络连接错误，使用tcpdump工具捕获和分析 TCP 数据包，并且联系最近公司换网可知，可能存在 WAF 或者防火墙的规则，对之前的 ip 地址匹配没问题，而现在的 ip 地址已更换，导致并不符合规则，进而被 WAF 或防火墙拦截，最终导致该 HTTP 请求网络链接错误。

  5. 后端同学联系运维同学，最后得知，是去年开发该提取余额接口，为了防止该接口被恶意访问或攻击，做了 WAF (应用层防火墙)限制，只要 HTTP 请求 url 等于该提取余额接口并且 ip 地址不属于配置的 ip 地址，就会被拦截掉，返回状态码 405。

    到此，可以知道接口为何报错跨域的原因了。

    在浏览器中，请求该接口，因为 WAF 拦截导致网络错误，而由第 3 步可知，浏览器响应（response）是网络错误会导致 HTTP 请求丢失Origin标头，所以浏览器控制台会报错提示跨域。

    可以验证，在 Postman 中进行接口请求，并手动加上Origin请求标头，接口会返回 405。

2、处理问题

运维同学将 WAF 针对该接口的规则进行修改，更新的 ip 白名单，最后问题解决。

三、了解 WAF 并实现 WAF demo

1、WAF 和网络防火墙的区别

WAF 是 Web Application Firewall 的缩写，也被称为 Web 应用防火墙，WAF 通过针对超文本传输协议 (HTTP) 流量来保护 Web 应用程序。这与标准防火墙不同，后者在外部和内部网络流量之间提供屏障。

网络防火墙可保护安全局域网 (LAN) 免受未经授权的访问，以防止攻击风险。其主要目标是将安全区域与不安全区域分隔开，并控制这两者之间的通信。如果没有网络防火墙，任何具有公共互联网协议 (IP) 地址的计算机都可以在网络外部被访问，并且可能面临攻击风险。这有助于防止的一些网络流量包括：

  • 未经授权的访问：攻击者未经许可访问网络。这种攻击常见的实现方式是，利用弱密码、社交工程或内部威胁来盗取凭据和入侵帐户。
  • 中间人 (MITM) 攻击：攻击者拦截网络和外部站点之间或网络内部的流量。造成这种攻击的常见原因是，不安全的通信协议使得攻击者能够在传输过程中窃取数据，然后获得用户凭据并劫持用户帐户。
  • 提权：攻击者可以访问网络，然后利用提权来扩大其在系统中的访问范围。提权细分为横向提权和纵向提权，前者可获得邻近系统的访问权限，后者可在同一个系统中获得更高权限。

WAF 位于外部用户和 Web 应用程序之间，以分析所有 HTTP 通信。然后，它会在恶意请求到达用户或 Web 应用程序之前对其进行检测和拦截。 因此，WAF 可以保护关键业务 Web 应用程序和 Web 服务器免受零日威胁及其他应用层攻击。这样可以保护企业免受针对应用程序的 Web 攻击包括：

  • 分布式拒绝服务：试图通过大量网络流量造成网络、服务或服务器不堪重负，使之因此而陷入瘫痪。这种攻击旨在耗尽目标的资源；由于流量的恶意性质并不总是明显的，因此可能难以防御。
  • SQL 注入：这种注入攻击使黑客可以执行恶意 SQL 语句，通过这些语句控制 Web 应用程序背后的数据库服务器。这使攻击者可以避开网页身份验证和授权并检索 SQL 数据库的内容，然后添加、修改和删除数据库的记录。网络罪犯可以利用 SQL 注入来访问客户信息、个人数据和知识产权。这种攻击在 2017 年 OWASP 十大安全威胁榜单中被列为第一位的 Web 应用安全威胁。
  • 跨站脚本：一种 Web 安全漏洞，使攻击者可以损害用户与应用程序的交互。它使攻击者可以规避同源策略（该策略旨在分隔不同的网站）。因此，攻击者可以伪装成真正的用户，并访问他们有权访问的数据和资源。

3、联系生产环境的问题，总结

上述 demo 更多为了了解一下 WAF 的使用，ModSecurity配合 nginx 实现 WAF。

而公司的 WAF 是阿里云的产品，联系生成环境出现的问题，出现HTTP 请求没有响应信息的情况，可能是匹配拦截规则后，执行了某个阻断动作(可能为drop)，断开了 TCP 连接，导致网络错误，进而浏览器请求丢失了*Origin请求标头，最终提示跨域的报错。
